# Phase 1 Spec：邮件扫描 → 分类 → 提取 → Excel 输出

> 本文档是 Phase 1「工作流验证」的完整规格说明。约定每个模块的输入、输出、边界条件和验收标准。

---

## 一、系统概览

### 1.1 模块流水线

```
[模块1: fetch_emails.py]        IMAP 连接 + 邮件获取
        ↓
[模块2: classify_emails.py]    域名匹配 + 关键词匹配双规则分类
        ↓
[模块3: extract_emails.py]     平台专用解析器 + 通用正则兜底提取
        ↓
[模块4: generate_report.py]    全链路串联 + Excel 报表 + 附件存档
```

### 1.2 设计约束

| 约束 | 说明 |
|------|------|
| 零 API Key | 所有模块基于规则引擎运行，不依赖任何 LLM |
| 本地处理 | 邮件数据不离开用户机器 |
| 只读 | 不修改邮箱内容 |
| 单次运行 | 用户主动触发，不做持续监控 |

## 二、模块1：邮件获取 (`fetch_emails.py`)

### 2.1 功能

连接 QQ 邮箱 IMAP，获取最近 N 封邮件的基本信息，验证连接可用性。

### 2.2 输入

| 输入项 | 来源 | 格式 |
|--------|------|------|
| QQ_EMAIL | `.env` | `user@qq.com` |
| QQ_AUTH_CODE | `.env` | QQ 邮箱 IMAP 授权码 |
| QQ_IMAP_SERVER | `.env`（可选，默认 `imap.qq.com`） | 主机名 |
| QQ_IMAP_PORT | `.env`（可选，默认 `993`） | 整数 |
| 邮件数量 | 函数参数 `count` | 默认 10 |

### 2.3 处理流程

1. 读取 `.env` 配置
2. 建立 IMAP SSL 连接
3. 登录认证
4. 选择 INBOX
5. 获取最近 `count` 封邮件
6. 解码并打印：发件人、主题、日期
7. 关闭连接

### 2.4 输出

**控制台**：
```
发件人: xxx@example.com
主题:   您的行程确认单
日期:   Mon, 30 Jun 2026 10:00:00 +0800
```

### 2.5 错误处理

| 场景 | 行为 |
|------|------|
| `.env` 未配置 | 打印提示 "请在 .env 文件中配置" |
| 登录失败 | 打印 `IMAP 错误: xxx` |
| 收件箱为空 | 打印 "收件箱为空或搜索失败" |
| 连接失败 | 打印 `连接失败: xxx` |

### 2.6 验收标准

- [ ] `.env` 配置正确时，能成功打印最近邮件列表
- [ ] 解码中文邮件标题不乱码
- [ ] 连接失败时有明确错误提示

## 三、模块2：邮件分类 (`classify_emails.py`)

### 3.1 功能

基于规则引擎，判断邮件是否与出差/旅行相关，并归类到具体品类。

### 3.2 分类规则

#### 3.2.1 域名匹配（优先级最高）

| 品类 | 域名（不区分大小写） |
|------|------|
| 机票 | `qunar.com`, `ctrip.com`, `fliggy.com`, `alitrip.com`, `airchina`, `ceair.com`, `csair.com`, `chinamobile.com`, `bookings.com`, `expedia`, `skyscanner`, `trip.com` |
| 火车票 | `12306.cn`, `zhixing.com`, `tiexing.com` |
| 酒店 | `booking.com`, `agoda.com`, `airbnb.com`, `meituan.com`, `dianping.com`, `elong.com` |
| 网约车 | `didiglobal.com`, `didipinduo.com`, `uber.com`, `gaode.com`, `shouqiev.com` |
| 发票 | `crestv.cn`, `fapiao.com`, `invoice.`（含此后缀）, `发票` |
| 门票/活动 | `damai.cn`, `maoyan.com`, `showstart.com` |

#### 3.2.2 关键词匹配（域名无命中时作为兜底）

| 品类 | 关键词（不区分大小写） |
|------|------|
| 机票 | 机票, 航班, 登机, 值机, 航空, flight, boarding |
| 火车票 | 火车票, 高铁, 动车, 12306, 车票, train |
| 酒店 | 酒店, 民宿, 入住, 预订成功, booking confirmation, hotel |
| 网约车 | 滴滴, 网约车, 行程, 快车, 专车, 出租车 |
| 发票 | 发票, invoice, 报销, 电子凭证, receipt, fapiao |
| 门票 | 门票, 演出, 景点, 展览, 入场券, ticket |

#### 3.2.3 黑名单（直接跳过）

| 域名关键词 |
|------|
| `job51`, `steampowered`, `email.apple.com`, `amazon`, `jd.com`, `taobao.com`, `tmall.com`, `pinduoduo`, `weixin`, `alipay` |

### 3.3 输入

| 输入项 | 来源 |
|--------|------|
| sender | 邮件 `From` 头（已解码） |
| subject | 邮件 `Subject` 头（已解码） |
| body | 邮件正文前 300 字符（text/plain） |

### 3.4 处理流程

```
1. 黑名单检查 → 命中则返回 "不相关"
2. 域名匹配 → 命中则返回对应品类 + "域名匹配"
3. 关键词匹配 → 命中则返回对应品类 + "关键词匹配"
4. 无命中 → 返回 "未识别" + "无匹配"
```

### 3.5 输出

每条邮件输出一个字典：

```python
{
    'is_relevant': True | False,
    'category': '机票' | '火车票' | '酒店' | '网约车' | '发票' | '门票/活动' | '不相关' | '未识别',
    'method': '域名匹配' | '关键词匹配' | '黑名单' | '无匹配'
}
```

### 3.6 验收标准

- [ ] 至少 5 大品类各覆盖 1 个平台
- [ ] 黑名单邮件不被误分类
- [ ] 域名匹配优先于关键词匹配
- [ ] 不区分大小写

## 四、模块3：结构化提取 (`extract_emails.py`)

### 4.1 功能

对已分类的邮件提取结构化字段。支持平台专用解析器 + 通用正则兜底。支持 PDF 附件文本提取。

### 4.2 提取优先级

| 优先顺序 | 数据源 |
|------|------|
| 1 | PDF 附件内容 |
| 2 | 邮件正文（text/plain） |
| 3 | 邮件正文（text/html → 清洗后纯文本） |
| 4 | 邮件主题行 |

### 4.3 平台专用解析器

每个解析器函数签名：`parse_{platform}(body: str, subject: str) -> dict`

#### 4.3.1 去哪儿网机票 (`parse_qunar_flight`)

| 提取字段 | 正则模式（按优先级） |
|------|------|
| 金额 | `[价额共].*?(\d+\.?\d*)` → `\d+\.\d{2}\s*元` |
| 订单号 | `订单[号编][码号]?[：:]\s*(\w+)` → `(订单|订).*?(\d{10,})` |
| 日期 | `\d{4}[-/年]\d{1,2}[-/月]\d{1,2}` → `出发[日期时间]*[：:]\s*\d{4}[-/年]\d{1,2}[-/月]\d{1,2}` |
| 出发地→目的地 | `([\u4e00-\u9fa5]{2,4})\s*[→\-]\s*([\u4e00-\u9fa5]{2,4})` |

默认字段：`平台='去哪儿网'`, `品类='机票'`

#### 4.3.2 滴滴出行 (`parse_didi`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `\d+\.?\d*\s*元` |
| 日期 | `\d{4}[-年]\d{1,2}[-月]\d{1,2}` |
| 出发地→目的地 | `[\u4e00-\u9fa5]{2,}[\u4e00-\u9fa50-9\-]*\s*[至→\-]\s*[\u4e00-\u9fa5]{2,}` |

默认字段：`平台='滴滴出行'`, `品类='网约车'`

#### 4.3.3 高德打车 (`parse_gaode`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `\d+\.?\d*\s*元` |
| 日期 | `\d{4}[-年]\d{1,2}[-月]\d{1,2}` |

默认字段：`平台='高德打车'`, `品类='网约车'`

#### 4.3.4 12306 火车票 (`parse_12306`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `\d+\.?\d*\s*元` |
| 日期 | `\d{4}[-年]\d{1,2}[-月]\d{1,2}` |
| 车次 | `[GCDZ]\d{1,6}` |

默认字段：`平台='12306'`, `品类='火车票'`

#### 4.3.5 华住酒店 (`parse_huazhu`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `\d+\.?\d*\s*元` |
| 日期 | `\d{4}[-年]\d{1,2}[-月]\d{1,2}` |
| 酒店名称 | `你好(.+?)酒店`（从主题） |
| 发票号码 | `发票号码[：:](\d+)`（从主题） |

默认字段：`平台='华住酒店'`, `品类='酒店'`

#### 4.3.6 智慧发票 (`parse_crestv`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `金额[：:](\d+\.?\d*)`（从主题）→ `\d+\.?\d*\s*元`（从正文） |
| 商家 | `【(.+?)】`（从主题） |
| 品类 | 含"油"→`发票(油费)`，含"餐"→`发票(餐饮)`，否则默认`发票` |

#### 4.3.7 票根 (`parse_txffp`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `发票金额[：:]\s*(\d+\.?\d*)` |
| 日期 | `\d{4}年\d{1,2}月\d{1,2}日` → `\d{4}[-年]\d{1,2}[-月]\d{1,2}` |
| 类型 | `收费公路通行费|高速[公费]?通行费` → `通行费` |

默认字段：`平台='票根'`, `品类='发票(通行费)'`

#### 4.3.8 携程机票 (`parse_ctrip_flight`)

| 提取字段 | 正则模式 |
|------|------|
| 金额 | `\d+\.?\d*\s*元` |
| 日期 | `\d{4}[-年]\d{1,2}[-月]\d{1,2}` |
| 出发地→目的地 | `([\u4e00-\u9fa5]{2,4})\s*[→\-]\s*([\u4e00-\u9fa5]{2,4})` |

默认字段：`平台='携程'`, `品类='机票'`

### 4.4 通用提取（兜底）

当邮件不属于任何已知平台时，使用 `generic_extract()`：

| 提取字段 | 正则模式（按优先级） |
|------|------|
| 金额 | `发票金额[：:]\s*(\d+\.?\d*)` → `金额[：:]\s*(\d+\.?\d*)` → `价[格款][：:]\s*(\d+\.?\d*)` → `合计[：:]\s*(\d+\.?\d*)` → `总计[：:]\s*(\d+\.?\d*)` → `¥\s*(\d+\.?\d*)` → `\d+\.\d{2}\s*元` → 主题中 `金额[：:](\d+\.?\d*)` |
| 日期 | `\d{4}年\d{1,2}月\d{1,2}日` → `\d{4}[-/年]\d{1,2}[-/月]\d{1,2}` |
| 订单号 | `(?:订单|发票)[号编][码号]?[：:（(]?\s*(\w{10,})` |

### 4.5 金额提取规范（全模块统一）

#### 4.5.1 优先级（从高到低）

| 优先级 | 来源 | 说明 |
|------|------|------|
| 1 | PDF 文件名中的金额 | 如 `1755.00.pdf`，直接从文件名末尾解析 |
| 2 | PDF 文本中的「价税合计(小写)」 | `价税合计[（(]小写[）)]\s*[¥￥](\d+\.?\d*)` |
| 3 | 正文中的「合计...元」 | `合[计]\s*(\d+\.?\d*)\s*元` |
| 4 | 正文中的 ¥/￥ 金额 | `[¥￥]\s*(\d+\.?\d*)` |
| 5 | 正文中的「XX.XX 元」 | `\d+\.\d{2}\s*元` |
| 6 | 兜底：正文中所有 XX.XX 数字中的最大值 | 仅当以上均无匹配时使用 |

#### 4.5.2 关键规则

- **禁止取税额**：`¥\s*(\d+)` 放在最后，避免匹配到发票中的税额（如 `¥ 2.81`）
- **禁止取 0-1 元区间**：兜底模式中排除 ≤1 的数值

### 4.6 PDF 附件处理

| 步骤 | 行为 |
|------|------|
| 1 | 遍历邮件所有 part，识别 `application/pdf` 类型或 `.pdf` 后缀附件 |
| 2 | 使用 PyPDF2 提取每页文本 |
| 3 | 空文本附件跳过 |
| 4 | 所有 PDF 文本合并到邮件正文末尾，用 `--- PDF附件 ---` 分隔 |

### 4.7 平台→解析器映射

```python
PLATFORM_PARSERS = {
    'qunar.com':       parse_qunar_flight,
    'xiaojukeji.com':  parse_didi,
    'didiglobal.com':  parse_didi,
    'crestv.cn':       parse_crestv,
    'huazhuhotels.com': parse_huazhu,
    'amap.com':        parse_gaode,
    '12306':           parse_12306,
    'txffp.com':       parse_txffp,
}
```

未匹配到的平台自动走 `generic_extract()`。

### 4.8 输出

每条邮件一个字典，必含字段：

```python
{
    '主题': str,      # 邮件原始主题
    '发件人': str,    # 解码后的发件人地址
    '分类': str,      # 品类名称
    '金额': float | None,
    '日期': str | None,
    # 以下为可选字段（视品类和平台而定）
    '出发地': str,
    '目的地': str,
    '订单号': str,
    '车次': str,
    '酒店名称': str,
    '商家': str,
    '类型': str,
    # ...
}
```

### 4.9 验收标准

- [ ] 8 个平台专用解析器全部能正确提取核心字段
- [ ] 金额提取不取税额（如高德行程单的 `¥ 2.81`）
- [ ] PDF 附件内容参与提取
- [ ] 新平台邮件能走通用提取流程，不会报错

## 五、模块4：全链路输出 (`generate_report.py`)

### 5.1 功能

串联模块1-3 全流程，生成 Excel 报表并保存附件。

### 5.2 输入

| 输入项 | 来源 | 说明 |
|--------|------|------|
| 邮箱配置 | `.env` | QQ_EMAIL, QQ_AUTH_CODE |
| 日期范围 | 交互式输入（可选） | `YYYY-MM-DD` 格式的开始和结束日期 |
| 扫描范围 | 逻辑确定 | 指定日期范围时按 IMAP SINCE/BEFORE 搜索，否则默认最近 60 封 |

### 5.3 日期范围过滤

```
用户输入开始日期 → 生成 SINCE DD-Mon-YYYY 搜索条件
用户输入结束日期 → 生成 BEFORE DD-Mon-YYYY 搜索条件
两者都不填 → 默认取最近 60 封邮件
```

### 5.4 处理流程

```
1. 连接邮箱
2. 询问日期范围 → 执行 IMAP 搜索
3. 遍历每封邮件：
   a. 分类（classify）
   b. 提取正文 + PDF/ZIP 内PDF文本
   c. 保存附件（PDF/ZIP/非广告图片）
   d. 如果是「机票」→ 按 PDF 附件拆分为多条独立记录
   e. 提取字段（extract_fields）
4. 汇总所有记录
5. 生成 Excel（3个Sheet）
6. 输出到 output/ 目录
```

### 5.5 机票按 PDF 附件拆分

| 规则 | 说明 |
|------|------|
| 触发条件 | 分类为「机票」 |
| 拆分方式 | 每个 PDF 附件生成一条独立记录 |
| 信息来源 | PDF 文件名解析日期、路线、金额（`\d{4}[-年]\d{1,2}[-月]\d{1,2}`, `[\u4e00-\u9fa5]{2,4}-[\u4e00-\u9fa5]{2,4}`, `\d+\.\d{2}\.pdf$`） |
| 不拆分 | 非机票品类按一封邮件一条记录处理 |

#### 机票 PDF 文件名解析示例

文件名格式：`2026-05-27 成都-甘孜-204482410162-机票电子发票-1755.00.pdf`

| 提取字段 | 来源 |
|------|------|
| 日期 | `2026-05-27` |
| 出发地 | `成都` |
| 目的地 | `甘孜` |
| 金额 | `1755.00` |

### 5.6 附件保存

| 保留类型 | 排除规则 |
|------|------|
| `.pdf` | — |
| `.zip` | — |
| `.jpg/.jpeg/.png/.gif/.bmp/.webp/.heic` | 排除文件名含 `advertising`, `logo`, `banner`, `icon`, `button`, `social`, `qrcode` 等广告/UI 素材 |
| 其他类型 | 丢弃 |

### 5.7 金额提取规范（generate_report.py 统一实现）

与模块3一致的五级优先级，另加：

- **PDF 文件信息覆盖**：若从 PDF 文件名已提取到金额/日期/路线，这些值直接覆盖正则提取结果
- **终极兜底**：若所有模式都未匹配，从正文中取所有 `XX.XX` 数值（排除 ≤1）的最大值

### 5.8 Excel 输出

**文件路径**：`output/差旅汇总_YYYYMMDD.xlsx`

#### Sheet 1：报销总览

| 区域 | 内容 |
|------|------|
| 标题行 | "差旅费用报销汇总单" |
| 报告日期行 | 生成日期 + 扫描范围 + 出差相关邮件数 |
| 总额卡片 | "可报销总额 ¥ X,XXX.XX"（蓝色大号字体） |
| 统计指标 | 费用笔数、单笔均额、分类数、无金额记录数 |
| 分类汇总表 | 类别、笔数、金额、占比（按金额降序）|
| 合计行 | 蓝色高亮合计行 |

**分类配色**：

| 品类 | 颜色 |
|------|------|
| 机票 | 蓝色 `DBEAFE` |
| 酒店 | 绿色 `D1FAE5` |
| 网约车 | 黄色 `FEF3C7` |
| 发票 | 粉色 `FCE7F3` |

#### Sheet 2：费用明细

| 出发地→目的地 | 路线，无则显示 `-` |
| 附件文件 | 关联的附件文件名 |

**排序**：按日期降序
**高亮**：最高金额行用红色加粗字体

#### Sheet 3：按供应商

| 列 | 说明 |
|------|------|
| 供应商 | 平台名称 |
| 笔数 | 该平台记录数 |
| 金额(元) | 汇总金额 |
| 占比 | 占总金额百分比 |

**排序**：按金额降序
**样式**：隔行灰白相间

### 5.9 验收标准

- [ ] 连接邮箱成功，能扫描邮件
- [ ] 指定日期范围过滤正确
- [ ] 机票邮件按 PDF 附件拆分为独立记录
- [ ] 附件保存到 `output/附件/`，广告图片被过滤
- [ ] Excel 三个 Sheet 内容完整、格式正确

- [ ] 无日期、无金额的记录不报错，显示 `-`

## 六、邮件解析通用规范

### 6.1 多编码支持

邮件头和正文按以下顺序尝试解码：

```
UTF-8 → GBK → GB2312 → GB18030 → UTF-8（errors='replace'）
```

### 6.2 正文提取顺序

```
1. text/plain（优先）
2. text/html（去标签后取纯文本，仅当无 text/plain 时）
3. PDF 附件文本（追加到正文末尾）
```

### 6.3 HTML 清洗

对 text/html 内容执行以下清洗后作为纯文本：

- 移除 `<style>` 和 `<script>` 标签及内容
- 移除所有 HTML 标签（替换为换行）
- 合并多余空行（3 个以上 → 2 个）
- 转换 `&nbsp;` 为空格

### 6.4 ZIP 附件处理

`generate_report.py` 额外支持 ZIP 附件中的 PDF：

1. 识别 `.zip` 后缀附件
2. 使用 `zipfile` 解压
3. 遍历内部文件，对 `.pdf` 文件提取文本
4. ZIP 自身不保存到附件目录

## 七、错误处理总则

| 场景 | 行为 |
|------|------|
| 邮箱连接失败 | 打印错误，退出 |
| 收件箱为空 | 打印提示 |
| PDF 解析异常 | 跳过该 PDF，继续处理 |
| ZIP 解压异常 | 跳过该 ZIP，继续处理 |
| 金额提取失败 | 记录为空字符串，不设默认值 |
| 无附件 | 附件字段留空 |

## 九、安全性规格

### 9.1 认证与凭证

| 规则 | 说明 |
|------|------|
| 凭证存储 | 邮箱账号和授权码仅存储在 `.env` 文件中 |
| Git 排除 | `.env` 已加入 `.gitignore`，不会被提交到版本库 |
| 传输加密 | IMAP 使用 SSL（端口 993），传输层加密 |
| 凭证作用域 | QQ 邮箱使用授权码而非登录密码，权限可控 |
| 只读操作 | 所有代码仅执行 `fetch` / `search`，不调用 `store` / `expunge` / `delete` |

### 9.2 数据本地化

| 规则 | 说明 |
|------|------|
| 数据不传出 | 邮件分类、提取、Excel 生成均在本地 Python 进程中完成 |
| 无日志泄露 | 邮件正文和提取字段仅打印摘要到控制台，不写入日志文件 |
| 无云存储 | 所有输出文件（Excel、附件）保存在本地 `output/` 目录 |

### 9.3 输出文件安全

| 规则 | 说明 |
|------|------|
| 敏感信息最小化 | Excel 报表仅包含行程和费用相关字段，不泄露邮件其他内容 |
| 附件格式白名单 | 仅保存 `.pdf`、`.zip`、非广告图片（`.jpg`/`.png` 等通过黑名单过滤） |
| 文件命名 | 附件以 `{序号}_{原始文件名}` 命名，不含邮箱地址 |

### 9.4 依赖安全

| 规则 | 说明 |
|------|------|
| 内置优先 | `imaplib`、`email`、`zipfile` 使用 Python 标准库 |
| 第三方最小化 | 仅 `python-dotenv`、`PyPDF2`、`openpyxl` 三个外部依赖 |
| 无执行风险 | 不调用 `eval()`、`exec()`，不使用 `subprocess` 执行外部命令 |
| 无网络请求 | 除 IMAP 连接外，代码不发任何外部 HTTP 请求 |

### 9.5 隐私设计

| 规则 | 说明 |
|------|------|
| 邮件原文不保存 | 运行结束后仅保留提取的结构化数据，不持久化原始邮件 |
| 附件关联可溯源 | 每条记录通过文件名关联原件，方便用户核对 |
| 不读取非差旅邮件 | 黑名单过滤（电商、社交、金融等），仅分析差旅相关邮件 |
| 用户可控 | 日期范围由用户指定，扫描范围用户决定 |

### 9.6 威胁模型与缓解

| 威胁 | 等级 | 缓解措施 |
|------|------|------|
| `.env` 误提交到 Git | 中 | `.gitignore` 排除 + 提交前 `git status` 检查 |
| 第三方依赖漏洞 | 低 | 依赖数量极少，定期 `pip list --outdated` 检查 |
| PDF 恶意文件 | 低 | PyPDF2 仅提取文本不解码 JavaScript/嵌入对象 |
| 控制台输出泄漏 | 低 | 仅打印摘要（主题前 40 字），不输出完整正文 |
| 附件文件覆盖 | 低 | 附件以序号+文件名命名，避免同名冲突 |
