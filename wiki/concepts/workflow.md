# BizTrip Agent 工作流设计

基于 Anthropic《构建有效 AI Agent》的架构模式设计。

## 设计原则

> **"成功不是构建最复杂的系统，而是构建适合需求的正确系统。"**
> — Anthropic

### 由简到繁的三层设计

```text
Level 1 (MVP)   → 提示链 + 路由
Level 2 (优化)   → 并行化
Level 3 (迭代)   → 评估器-优化器 (Phase 2)
```

每一层都只在上层证明不够用时才增加。

---

## Level 1：提示链 + 路由（MVP）

### 整体流程

```text
[Step 1] 获取邮件
  IMAP 连接 → 获取最近 60 天邮件
  Gate: 连接是否成功？
  失败 → 报错退出
  成功 → 进入 Step 2

[Step 2] 路由分类
  对每封邮件：
    LLM 判断：是否与出差相关？
    否 → 跳过
    是 → 细分品类：
      ├── 机票     → ［专用提取 prompt］
      ├── 火车票   → ［专用提取 prompt］
      ├── 酒店     → ［专用提取 prompt］
      ├── 网约车   → ［专用提取 prompt］
      ├── 油费单   → ［专用提取 prompt］
      └── 发票     → ［专用提取 prompt］
  Gate: 至少提取到一条有效记录？
  失败 → 提示用户"未找到出差相关邮件"
  成功 → 进入 Step 3

[Step 3] 出差聚合
  LLM 分析所有提取的记录：
    哪些属于同一次出差？
    （按同一时间段 + 同一目的地的订单聚合）
  输出：按出差行程分组的结构化数据

[Step 4] 生成 Excel
  按模板生成多 Sheet 的 .xlsx：
  - 出差行程总表
  - 机票明细（多人按人拆分）
  - 火车票明细
  - 酒店明细
  - 网约车明细
  - 油费/发票明细
  - 报销汇总表

[Step 5] 下载原件
  自动下载邮件中的 PDF 附件
  与提取的结构化数据关联存储
```

## Level 2：并行化（优化）

当 Level 1 跑通后，如果处理 60 天的邮件耗时过长，增加并行化：

```text
[Step 2 改为并行]
  邮件列表 → 分成 N 批次
  ├── 批次1 → LLM 并行分类+提取
  ├── 批次2 → LLM 并行分类+提取
  └── 批次N → LLM 并行分类+提取
  → 合并结果 → 进入 Step 3
```

### 适用条件
- 邮件量 > 50 封
- 单次处理时间 > 30 秒
- 用户可接受 N 倍的 API 成本换取速度

## Level 3：评估器-优化器（Phase 2）

当 Phase 2 Web 应用上线后，加入用户反馈闭环：

```text
用户看到提取结果 → 手动修正错误 → 系统记录修正模式
→ 定期用修正数据优化 prompt
→ 下次同类邮件准确率提升
```

参考文章的 "评估器-优化器" 模式：

```python
extraction_evaluation_prompt = """
评估以下信息提取的质量。

邮件原文：{email_body}
AI 提取结果：{extraction_result}
用户修正后的结果：{user_correction}

评估维度：
1. 字段完整性：缺失了哪些字段？
2. 字段准确性：哪些字段提取错误？
3. 分类准确性：品类判断是否正确？

请给出改进 prompt 的具体建议。
"""
```

## 模式选择依据

参考文章的决策流程图：

| BizTrip Agent 的判断 | 选择 | 原因 |
|--------------|------|------|
| 任务可分解为固定步骤吗？ | ✅ → 提示链 | 获取→分类→提取→聚合→输出，步骤固定 |
| 输入需要分类处理吗？ | ✅ → 路由 | 不同品类需要不同的提取 prompt |
| 子任务可并行吗？ | ✅ → 并行化（Level 2） | 多封邮件可同时处理 |
| 需要迭代优化吗？ | ⚠️ → 评估器-优化器（Phase 2） | 目前不需要，Phase 2 加用户反馈后用 |
